# Decisions

## ADR-001: No Database

Decision: use local JSON files for the first version.

Rationale:

- The requested product scope is a local session console, not a multi-tenant persistent service.
- It keeps the deployment simple.
- It avoids duplicating AgentScope state.

Consequences:

- Store methods use synchronized access and atomic file replacement.
- Cross-process concurrent writes are not strongly coordinated.
- Future database introduction should be a deliberate architecture change, not an incidental dependency addition.

## ADR-002: UI JSON Is Projection Only

Decision: `ui-store` files exist for display, listing, summary, and trace panels only.

Rationale:

- Agent inference state should come from AgentScope Java state and memory, not from replaying UI messages.
- This prevents accidental cross-session prompt stitching.

Consequences:

- Do not use `messages/{sessionId}.json` as the prompt source.
- Do not rebuild memory from UI JSON.

## ADR-003: AgentScope Owns Runtime State

Decision: `ReActAgent`, memory, session persistence, and execution loop are delegated to AgentScope Java.

Rationale:

- The explicit project goal is to avoid rebuilding AgentScope Java runtime capabilities.

Consequences:

- `AgentScopeRuntimeService` should remain a thin adapter.
- Avoid adding orchestration classes that recreate ReAct behavior.

## ADR-004: Fresh Agent And Memory Per Call

Decision: create a fresh `ReActAgent` and fresh `Memory` for every chat call, then load/save only the current AgentScope session.

Rationale:

- This keeps session state isolation explicit and avoids global mutable runtime state.

Consequences:

- Shared model clients are acceptable only if they are stateless/thread-safe.
- Static/global memory is forbidden.

## ADR-005: AgentScope Session ID Equals Frontend Session ID

Decision: `agentscopeSessionId = session.id`.

Rationale:

- Simple one-to-one mapping.
- Easy to inspect local storage.
- Session IDs are generated by backend, not user input.

Consequences:

- Session ID validation remains important before calling `JsonSession`.
- If future migrations need different IDs, preserve backward compatibility in `ChatSessionDTO.agentscopeSessionId`.

## ADR-006: Trace Is A Placeholder In V1

Decision: create trace JSON documents and endpoints now, but leave items empty until real AgentScope telemetry mapping is implemented.

Rationale:

- The UI can stabilize around the Trace tab without inventing a fake tracing model.

Consequences:

- Do not fabricate spans to make the UI look active.
- Future implementation should map official AgentScope telemetry/tracer output.

## ADR-007: Default Provider Currently Ollama

Decision observed in current code/config: default model provider is `ollama`.

Rationale:

- Current `application.yml` is configured for local Ollama and uses `OPENAI_API_KEY:not-needed`.

Consequences:

- DashScope API-key behavior exists but is not the default path.
- Update docs/config carefully if switching back to DashScope.

## ADR-008: Runtime State Lives Under Project `.agents`

Decision: local runtime state moved from `${user.home}/.agent-console` to project-root `.agents` subdirectories: `${user.dir}/.agents/ui-store` and `${user.dir}/.agents/agentscope-sessions`.

Rationale:

- Keeps this app's local state colocated with the project.
- Makes handoff and cleanup easier.
- Allows `.agents/skills`, `.agents/ui-store`, and `.agents/agentscope-sessions` to share one project-local root while still separating source-like skills from runtime state.

Consequences:

- `.agents/ui-store/` and `.agents/agentscope-sessions/` are ignored by git.
- `.agents/skills/` remains project-local configuration and should not be treated as runtime output.
